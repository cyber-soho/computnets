Concept Primer

02Know These Before the CLI

Click each concept to expand its explanation. These are not definitions to memorize — they are mental models you will need while you work.

📦
The Packet
The atom of networking
+
Everything that travels across a network does so in packets — small, discrete units of data. Think of a packet like a postcard: it has a destination address, it travels through several "post offices" (routers), and each router decides where to forward it next.

Each packet contains a header (source IP, destination IP, protocol type, TTL — Time To Live), a payload (the actual data), and sometimes a trailer (error-checking information).
TTL (Time To Live) is a counter. Each router that forwards the packet decrements it by 1. If it reaches 0, the packet is discarded. This prevents lost packets from circling the internet forever.
🔍
ARP
Address Resolution Protocol
+
Your computer thinks in IP addresses (e.g., 192.168.10.5), but the physical network hardware thinks in MAC (Media Access Control) addresses (e.g., AA:BB:CC:DD:EE:FF). ARP is the bridge between these two worlds.

When your computer wants to reach another device on the same local segment, it broadcasts (shouts to everyone): "Who has IP 192.168.10.1? Tell me your MAC address." The correct device replies with its MAC. Your computer caches this mapping in its ARP cache for a few minutes.
⚠️ ARP has no authentication. A malicious device can send fake ARP replies to redirect traffic — this is called ARP Spoofing / ARP Poisoning. Understanding ARP is the first step to understanding this attack and its defenses (Dynamic ARP Inspection, static ARP entries).
📡
ICMP
Internet Control Message Protocol
+
ICMP is the "voice" of the network — it carries error messages and operational information between devices. When a router cannot forward a packet, it sends an ICMP message back to the sender explaining why.

TypeNameMeaning
Type 8Echo RequestSent by ping — "Are you there?"
Type 0Echo ReplyResponse to ping — "Yes, I'm here."
Type 3Dest. UnreachableRouter cannot reach destination
Type 11Time ExceededTTL hit zero — used by traceroute
🔔
Ping
The simplest reachability test
+
ping sends ICMP Echo Requests to a destination and reports whether replies were received, the RTT (Round-Trip Time) in milliseconds, and packet loss percentage.

A successful ping tells you: a packet can travel from here to there and back. A failed ping tells you something is blocking the path — but not where. That is what traceroute is for.
Ping is the "can you hear me?" of networking. It is usually the first diagnostic tool a network engineer reaches for.
🗺️
Traceroute / Tracert
Following the packet's path
+
traceroute (Linux/macOS) or tracert (Windows) sends packets with incrementally increasing TTL values. The first packet has TTL=1, so the first router decrements it to 0 and sends back an ICMP Time Exceeded message — revealing its own address. Each subsequent packet travels one hop further.

Result: a hop-by-hop map of every router the packet crosses, with latency at each hop.
traceroute 10.10.30.1
1  10.10.10.1  1ms  — default gateway
2  10.10.20.1  2ms  — core router
3  10.10.30.1  3ms  — destination
💥
Fault Injection
Why chaos is your friend
+
A network engineer who has only ever tested a working network is like a pilot who has only ever flown in perfect weather. Deliberately breaking things in simulation teaches you which redundant paths exist, how fast the network detects failures, and which subnets lose reachability.

This discipline is called chaos engineering in professional contexts. Organizations like Netflix (Chaos Monkey) have built entire engineering practices around deliberately injecting failures into production systems to discover weaknesses before customers do.
🦈
Wireshark
Packet analyzer / protocol sniffer
+
Wireshark is a free, open-source packet analyzer that captures all network traffic passing through an interface and displays it decoded layer by layer. It lets you see every ARP broadcast, every ICMP exchange, every TCP handshake — in real time.

Key display filters:
arp  → show only ARP frames
icmp  → show only ICMP packets
ip.addr == 10.10.10.5  → filter by IP

⚠️ Capturing traffic not destined for your machine requires promiscuous mode (NIC — Network Interface Card — captures all packets on the segment, not just its own) and may be legally regulated. Always use only on authorized networks.
🔧 Tool Freedom Reminder: Wireshark is a suggestion. tcpdump, Scapy, Zeek, or a custom Python script can all capture and analyze packets. Research your options. Consider building your own. The tool is secondary — the understanding is primary.
Lab Procedure

03Step-by-Step Lab

Follow the parts in order. The environment assumed is a network simulator — adapt all commands to your setup.

A1
Verify Routing Tables

On every router in your topology, display the routing table. For every subnet that should exist, confirm there is a route to it, the next-hop address makes sense, and the outgoing interface is correct.

Cisco IOS CLI 
Router# show ip route

-- Example output excerpt --
C   10.10.10.0/24 is directly connected, Gig0/0.10
C   10.10.20.0/24 is directly connected, Gig0/0.20
S   10.20.0.0/24 [1/0] via 203.0.113.2

Document: Screenshot or copy-paste the routing table of every router. Label each one clearly by hostname.

Build a Ping Connectivity Matrix

Create a table with every subnet as both source and destination. Systematically ping every combination and mark each cell ✓ (reachable) or ✗ (not reachable). Note the RTT (Round-Trip Time) for successful pings.

Run Traceroute for At Least 3 Paths

Pick at least three paths that cross a router boundary. Record complete output — every hop, every address. Annotate each hop: what device is it? What interface? Why does the packet go that way?

Before you break anything: write your hypothesis for each scenario first. Engineering is structured inquiry, not random experimentation.

F-1
Shut down the Access-to-Core uplink
▼
  • Write hypothesis: which subnets do you predict will lose connectivity?
  • Apply: interface GigX/X → shutdown
  • Re-run your ping matrix — mark which cells change from ✓ to ✗
  • Run traceroute to an affected destination — note the * * * timeouts
  • Capture in Wireshark — look for ICMP Destination Unreachable (Type 3) messages
  • Restore: no shutdown → confirm baseline returns
  • Record: was your hypothesis correct? If not, why not?
F-2
Shut down one VLAN sub-interface on Router-on-a-Stick
▼
  • Write hypothesis: which VLAN loses inter-VLAN routing? Which are unaffected?
  • Apply: interface Gig0/0.20 → shutdown
  • Re-run ping matrix — note that only one VLAN should be affected
  • Verify: can devices in the disabled VLAN still reach other devices in the same VLAN?
  • Check the routing table with show ip route — what changed?
  • Restore and verify
Cisco IOS CLI
Router(config)# interface GigabitEthernet0/0.20
Router(config-subif)# shutdown
-- test -- then restore:
Router(config-subif)# no shutdown
F-3
Disconnect the WAN link to the Branch
▼
  • Write hypothesis: which subnets at HQ can still communicate? What happens to Branch reachability?
  • Apply: shut down or disconnect the serial / WAN interface connecting HQ to the Branch
  • Re-run the full ping matrix — note which rows/columns now show ✗
  • Does HQ-to-HQ communication remain fully functional?
  • What ICMP message type does the HQ router send back to a pinging host?
  • Document whether your network has a redundant WAN path — if not, this is a single point of failure
  • Restore and verify

Failure Analysis (include in report)

After testing all scenarios: Which parts of your topology have single points of failure? What redundancy improvements would you recommend for a real organization? Reference the Cisco High Availability Design Guide or equivalent documentation in your analysis.

C1
ARP Dissection

Capture a fresh ARP exchange. Expand the ARP layer in your analyzer. Answer in your report:

What is the destination MAC of the ARP Request? Why is it FF:FF:FF:FF:FF:FF?
What fields identify the sender and target in the ARP packet?
Run arp -a on a host after a successful ping — what do you see in the ARP cache?
C2
ICMP Dissection

Capture a ping session (4 packets) across a routed boundary. Answer:

What is the TTL (Time To Live) value? How does it change at each hop?
What is the ICMP Identifier field? Why might it be useful?
Compare the RTT (Round-Trip Time) shown by ping to the timestamps in Wireshark — are they consistent?
C3
Failure Traffic Analysis

With a link shut down, initiate a ping to an unreachable host. Capture the traffic. Answer:

What ICMP message type does the router generate to inform the sender?
What information is included in the Data section of that ICMP error message?
Can you see ARP requests even when the destination is unreachable? What does this tell you about the network stack's order of operations?
D1
IANA Registry Research

The IANA (Internet Assigned Numbers Authority) maintains the official registry of all ICMP message types.

Browse the registry. Find 3 ICMP message types you have not yet encountered in this lab. For each, write two sentences: what network event generates this message, and why it would be useful to a network engineer.
